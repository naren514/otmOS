Definitions - ITINERARY

This table defines the path between two locations. The itinerary is described in terms of the legs. An itinerary can have one or more legs. The itinerary table consists of the constraints on the weight allowed for shipment for each path. The itinerary can be used by a corporation or can specify additional constraints using the calendar GID like Monday-Friday Itinerary, Weekend Itinerary etc.

ITINERARY_GID
ITINERARY_GID contains the unique identifier for the intinerary.
ITINERARY_XID
ITINERARY_XID contains the unique external identifier for the itinerary.
ITINERARY_NAME
ITINERARY_NAME contains the name for the itinerary.
DESCRIPTION
DESCRIPTION contains text describing the itinerary.
MIN_WEIGHT
MIN_WEIGHT contains the minimum weight for the shipment on the itinerary.
MIN_WEIGHT_UOM_CODE

MIN_WEIGHT_BASE
MIN_WEIGHT in base units.
MAX_WEIGHT
MAX_WEIGHT contains the maximum weight for the shipment on the itinerary.
MAX_WEIGHT_UOM_CODE
Unit of measure for MAX_WEIGHT column.
MAX_WEIGHT_BASE
MAX_WEIGHT in base units.

MIN_VOLUME contains the minimum volume for the shipment on the itinerary.

Unit of measure for MIN_VOLUME column.
MIN_VOLUME_BASE

MAX_VOLUME
MAX_VOLUME contains the maximum volume for the shipment on the itinerary.
MAX_VOLUME_UOM_CODE
Unit of measure for MAX_VOLUME column.
MAX_VOLUME_BASE
MAX_VOLUME in base units.
CORPORATION_PROFILE_GID

CALENDAR_GID
CALENDAR_GID contains the unique identifier for the calendar.

IS_MULTI_STOP describes whether the itinerary contains more than one stop. IF so, this column contains a Y for yes; if not it contains an N for no.

USE_CONSOLIDATION_POOL

MIN_STOPOFF_WEIGHT
order bundles less than this weight will always go through a pool if the orgin/dest of the order bundles fall into consolidation/de-consolidation pools
MIN_STOPOFF_WEIGHT_UOM_CODE
Unit of measure for MIN_STOPOFF_WEIGHT column.
MIN_STOPOFF_WEIGHT_BASE
MIN_STOPOFF_WEIGHT in base units.
MAX_POOL_WEIGHT_BASE
MAX_POOL_WEIGHT in base units.
TOTAL_STOPS_CONSTRAINT
: it specifies the maximum number of stops on a multi-stop shipment derived from this itinerary.
MIN_TL_WEIGHT
: This is the minimum weight to make up a nice truckload. It is also used as the minimum weight to go through a pool
MIN_TL_WEIGHT_UOM_CODE

MIN_TL_WEIGHT_BASE
MIN_TL_WEIGHT in base units.
MIN_STOPOFF_VOLUME
Order bundles less than this volume will always go through a pool if the origin/dest of the order bundles fall into consolidation/de-consolidation pools. When doing pool drop swapping, only pool drop shipments whose volume is bigger than this value are tried.
MIN_STOPOFF_VOLUME_UOM_CODE
Unit of measure for the min_stopoff_volume.
MIN_STOPOFF_VOLUME_BASE
Stopoff volume in base units.
MAX_POOL_VOLUME
This is the maximum volume to go through a pool.
MAX_POOL_VOLUME_UOM_CODE
Unit of measure for the maximum pool volume.
MAX_POOL_VOLUME_BASE
Maximum volume to go through a pool in base units.
MAX_SMALL_DIRECT_VOLUME
After pool swapping, we try to pull off the orders that are close to parcel weight and ship direct, if the overall solution is cheaper. This volume guides the algorithm to only pull of the one that is smaller than this volume.
MAX_SMALL_DIRECT_VOL_UOM_CODE
Unit of measure for the small direct volume.
MAX_SMALL_DIRECT_VOLUME_BASE
Small direct volume in base units.
MAX_SMALL_DIRECT_WEIGHT
After pool swapping, we try to pull off the orders that are close to parcel weight and ship direct, if the overall solution is cheaper. This weight guides the algorithm to only pull of the one that is smaller than this weight.
MAX_SMALL_DIRECT_WGHT_UOM_CODE
Unit of measure for small direct weight.
MAX_SMALL_DIRECT_WEIGHT_BASE
Small direct weight in base units.
MIN_TL_USAGE_PERCENTAGE
The percentage of usage to be used to determine whether a TL is full in bulk plan. This will guide whether direct shipment be kept without routing to pool. It also determines whether pool shipments be kept without routing to xdock.
IS_DEST_BUNDLE_PREFERED
If this flag is set, when going through the de-consolidation pool process, if there is a x-dock on this itinerary, two or more order bundles have the same destination will not route to the de-consolidation pool. It will route to x-dock later for consolidation at x-dock and direct go to the destination
HAZMAT_MODE_GID
A unique identifier to define the transportation modes. (includes the domain name as part of the key)
ITINERARY_TYPE
When 'N', indicates itinerary is inactive and should not be used by the algorithm.
PICKUP_STOPS_CONSTRAINTS
total # of pickup stops for a multistop shipment. It is usually set to be the max of PICKUP_STOPS_CONSTRAINTS of all valid rate offering. This help multistop performance
DELIVERY_STOPS_CONSTRAINTS
total # of pickup stops for a multistop shipment. It is usually set to be the max of DELIVERY_STOPS_CONSTRAINTS of all valid rate offering. This help multistop performance
MAX_DIST_BTW_PICKUPS
if distance between 2 pickup stops is beyond this, don't pair 2 shipments in multistop
RADIUS_FOR_PICKUPS
the distance from any pickup stop to the root bundle's source location can not exceed this field.
RADIUS_PERCENTAGE_FOR_PICKUPS
distance from any pickup to the root source divided by the root distance can not exceed this planning parameter.
PAYMENT_METHOD_PROFILE_GID
List of compatible payment methods for the itinerary
